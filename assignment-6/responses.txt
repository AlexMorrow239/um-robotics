Path planning is the concept of navigating from some initial point to some goal point whilst avoiding obstacles along the way. Path planning can also involve optimizing some cost function to find the ideal path through the space.

The basic steps of the RRT algorithm are as follows:

- from x(init), select a random point x(rand)
- find the nearest vertex, x(near), to x(rand) in T
- compute x(new) from these points such that x(near)x(new) is a subset of c(free). In other words, all points along the path from x(near) to x(new) or unobstructed.
- occasionally set x(rand) to the goal position as well.

RRT is known to find solutions to path planning problems quickly, but the solution itself may be poor because RRT can get stuck following a poor path based on its initial direction of exploration. RRT* attempts to fix this flaw by using local label correction.

Implementation Explanation:

My implementation generally follows the steps outlined above. I also implemented a quick_path function which checks if a direct line path from start to goal exists and is obstacle free. If so, the robot will use this path instead of going through the entire RRT planner. I increased the obstacle inflation from 0.9 to 5.0 grid cells to ensure the robot does not scrape by or get stopped by an obstacle in its planned path. I explored more sophisticated kinds of obstacle detection to account for this, but I achieve basically the same result by using simple inflation, so it was not needed. I also added dynamic obstacle detection and path replanning. If the robot detects an obstacle directly in front of itself, it will stop and replan its route tot he goal from its current position using rrt again. Another challenge that I ran into had to do with publishing the nodes and tree visualizations. I kept running into issues with ID overlaps and dissonance between the nodes and the corresponding tree. To solve this, I created large constants to define ID ranges for individual rviz components. This ensured that all the marker objects within the provided publisher would always have unique ID's. As part of this, I also added a clear tree function that would remove all visualization components from previous navigation tasks. I also found that the occupancy_grid naming was a bit confusing. There was a possibility of passing the already inflated grid to a function that was anticipating a non_inflated grid. I addressed this by just changing the variable names for additional clarity. I also think it would be better if many of these functions relied on self. notation instead of expecting attributes of itself to be repassed as function arguments. This would reduce the chances of making a mistake by passing soething incorrectly.
